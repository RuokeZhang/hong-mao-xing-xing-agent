# 写一个与红毛猩猩结合的agent项目

        ## Insights
        - 把“红毛猩猩”当作统一人格壳：底层可以是多 Agent（分析、心理、历史），但对用户只暴露一个红毛猩猩角色，体验更完整也更易讲故事和展示
- 在 60 小时内，优先做“好用 + 有趣”的 MVP，而不是追求复杂的医疗级健康监测或跨平台客户端；Next.js 单页 + 简单多 Agent 调度已经足够亮眼
- 可以借鉴 Healthcare LAMAgents 的多 Agent分工思路，但将数据源从硬件健康指标简化为“文本情绪 + 话题”，大幅降低集成成本
- 红毛猩猩元素不必只体现在 UI，可以深入到对话策略：例如用“森林栖息地是否健康”的隐喻来帮助用户理解自己的情绪与习惯
- 用可视化情绪反馈（小表情/颜色条）既能展示 AnalystAgent 的价值，又不会增加太多前端复杂度，是 hackathon 演示时的记忆点

        ## Warnings
        - 避免把项目做成“医疗诊断”或“专业心理治疗”工具，文案中要明确是情绪陪伴/反思助手，减少合规与伦理风险
- 不要在 60 小时内尝试接入可穿戴设备、Twilio、医院急救等重集成能力，这会严重挤占时间并增加不稳定因素
- 多 Agent 调度逻辑要保持简单（例如固定调用顺序），过度追求通用 Planner/Caller 框架容易导致调试时间爆炸
- 心理学相关输出要避免给出具体药物、诊断或强指令式建议，更多使用引导式问题和普适性建议
- 如果使用第三方 LLM API，要注意对话与情绪数据的隐私提示，避免在演示中上传真实敏感信息
